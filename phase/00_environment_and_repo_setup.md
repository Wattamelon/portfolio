# Phase 00. GitHub와 Docker 기반 작업 환경 세팅

## 이 단계의 목표
이 단계에서는 앞으로의 모든 개발이 흔들리지 않도록, 저장소와 실행 환경의 기준을 먼저 잡습니다.

특히 이 프로젝트는 처음에는 개인 맥북에서 작업하다가 나중에 iMac으로 옮겨야 하므로,
특정 컴퓨터에만 묶이는 방식이 아니라 **어디서 열어도 같은 방식으로 실행되는 구조**를 목표로 합니다.

## 미션에서 충족할 요구
- GitHub 저장소 기반으로 버전 관리 시작
- 로컬 개발 환경과 원격 저장소의 흐름 정리
- Docker를 활용한 실행 환경 표준화
- 컴퓨터가 바뀌어도 동일하게 열 수 있는 프로젝트 구조 마련
- 이후 phase 작업들이 안정적으로 이어질 수 있는 기반 만들기

## Codex에게 요청할 작업
1. Git 저장소가 없다면 초기화하고, 기본 브랜치 전략을 정리한다.
2. GitHub 원격 저장소를 연결할 수 있도록 준비한다.
3. `.gitignore`를 만들어 OS 파일, 에디터 파일, 빌드 산출물, 환경 파일을 제외한다.
4. 필요하다면 `.env.example`을 만들어 환경 변수 예시를 분리한다.
5. 프로젝트가 정적 웹사이트라면 Docker로 정적 서버를 띄우는 구성을 만든다.
6. 로컬 개발과 Docker 실행 방식이 충돌하지 않도록 문서를 정리한다.
7. README 초안을 만들어 실행 방법, 개발 방식, 배포 흐름의 뼈대를 적는다.

## 왜 이 단계가 필요한가
처음부터 환경을 정리해두면, 나중에 컴퓨터를 옮겨도 작업이 덜 꼬입니다.
GitHub는 버전과 히스토리를 지켜주고, Docker는 실행 환경을 비슷하게 맞춰줍니다.

즉,
- GitHub는 "코드가 어디에 있고 어떻게 기록되는가"를 해결하고
- Docker는 "어느 컴퓨터에서 실행해도 같은가"를 해결합니다.

## 이 단계가 끝나면 나오는 결과
- GitHub로 연결된 버전 관리 구조
- 불필요한 파일이 커밋되지 않는 깔끔한 저장소
- Docker로 실행할 수 있는 표준 환경
- 맥북에서 시작해 iMac으로 옮겨도 같은 흐름으로 작업할 수 있는 기반

## 완료 기준
- 저장소가 Git으로 관리된다.
- 원격 GitHub 저장소와 연결할 준비가 되어 있다.
- `.gitignore`가 정리되어 있다.
- Docker 실행 방식이 문서화되어 있다.
- 다른 기기에서도 동일한 실행 절차를 따라갈 수 있다.

## Codex 출력 기대값
- GitHub와 Docker를 고려한 초기 프로젝트 세팅
- 이후 phase가 흔들리지 않도록 정리된 실행 기준
